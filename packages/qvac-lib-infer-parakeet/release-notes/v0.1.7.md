# QVAC Transcription Parakeet Addon v0.1.7 Release Notes

This release adds native C++ support for loading ONNX sessions directly from individual file paths, enabling the SDK to resolve model artifacts independently from registries, HTTP, or local storage without requiring all files to reside in a single directory.

## New Features

### Direct file path loading in the native addon

The C++ addon now accepts individual file paths (`encoderPath`, `encoderDataPath`, `decoderPath`, `vocabPath`, `preprocessorPath`) via the JS configuration object. When provided, ONNX sessions are created directly from these paths instead of going through buffer-based weight streaming.

```js
const config = {
  modelPath: '/fallback/model/dir',
  encoderPath: '/cache/abc123_encoder-model.onnx',
  encoderDataPath: '/cache/def456_encoder-model.onnx.data',
  decoderPath: '/cache/ghi789_decoder_joint-model.onnx',
  vocabPath: '/cache/jkl012_vocab.txt',
  preprocessorPath: '/cache/mno345_preprocessor.onnx',
  modelType: 'tdt',
}
```

This follows the same pattern as `@qvac/tts-onnx` and enables the SDK `resolveModelPath` to resolve each file independently (supporting `registry://`, `http://`, `pear://`, and local paths).

### Encoder external data staging

For encoder models with external `.onnx.data` files, the addon creates a temporary staging directory with symlinks using canonical filenames so ONNX Runtime can resolve the external data relative to the model path. The staging directory is cleaned up immediately after session creation.

### Vocabulary loading from file path

When named paths are provided and the vocabulary has not been loaded via `set_weights_for_file`, the addon reads the vocabulary directly from `vocabPath`.

## JavaScript changes

- When named paths are detected, `_load()` passes them to the native addon configuration and **skips** buffer-based `_loadModelWeights`, reducing memory overhead for large models.
- `_downloadWeights()` short-circuits when named paths are provided, since the SDK has already resolved files to disk.

## Backward Compatibility

When named paths are not provided, all existing behavior is preserved:
- Files are loaded via buffer streaming through `set_weights_for_file` / `_loadModelWeights`
- The encoder external data is resolved from `modelPath` on disk
- No migration required for existing callers
